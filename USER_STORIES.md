# USER_STORIES.md — Pixel Snatter 用户故事集

> 配套 [PLAN.md](./PLAN.md)。PLAN 回答"技术怎么做"，本文档回答"用户为什么用、用成什么样算成功"。

> 定位（来自需求澄清）：**双形态（CLI + Web）并重**，服务**独立游戏开发者 + 像素美术/设计师**，**完整产品**边界（含预设/会话/调色板编辑器等）。

- 🔴 **MVP** — 可用性完备的最小必需集。缺一项则核心闭环不成立。

## 一、Persona（用户画像）

### P1 — 独立游戏开发者「Kai」
- 一个人或小团队做游戏，AI 生成大量角色/道具/tile sprite，要快速清洗成引擎可用资产。
- **同时用 CLI 和 Web**：Web 调单图参数，确定后用 CLI 批量跑整个素材目录进 CI。
- 最看重：**确定性**（资产可复现，CI 不抖动）、**批量**、**引擎友好输出**（透明背景、描边、固定调色板、x2/x4 缩放导出）、**脚本可集成**。
- 痛点：AI 图网格漂移、颜色杂、有背景、尺寸不一。

### P2 — 像素美术/设计师「Mei」
- 接 AI 辅助产出但要精修交付。主要用 **Web**，几乎不用 CLI。
- 最看重：**即时预览**、**候选网格可选**、**调色板精细控制**（主机调色板 + 自定义 + dithering）、**滑块/放大镜逐像素对比**、**预设复用**。
- 痛点：参数太多不会调、要看效果才敢定、要反复对比。

---

## 二、MVP 定义（可用性完备最小集）

> 以下 🔴 story 构成"能交付"的底线：两类用户都能完成 **AI 脏图 → 引擎/美术可用资产** 的核心闭环，单图 + 批量，CLI + Web 双形态，确定性保证。

MVP 不含：矢量化、dithering、描边、形态学、预设管理 UI、会话历史、调色板编辑器（这些是 🟡/🟢）。

## 四、双形态对照（关键 story 的 CLI vs Web 表现）

| 能力 | CLI 表现 | Web 表现 |
|------|---------|----------|
| 输入 | 文件/目录路径参数 | 拖拽/粘贴/选择 |
| 参数 | flags + `--preset file` + `--json` | 表单控件 + 实时预览 |
| 网格候选 | `--json` 输出候选列表供脚本选 | 可视候选缩略图点击 |
| 对比 | 无（脚本看文件） | 滑块 + 放大镜 |
| 批量 | 目录 + `--skip-existing` + 并行 | 多图会话 + ZIP 导出 |
| 预设 | `.json` 文件，CI 友好 | localStorage + 导入导出 |
| 配方追溯 | sidecar `.json` | PNG embed 或 history |
| 输出 | 写文件（PNG，按 stem 命名） | 下载/复制到剪贴板 |

## 五、横切关注（非 story，但影响所有 story）

- **性能**：Web Worker 非阻塞（U12.5）；CLI rayon 并行批量；大图（>4K）单图 < 2s 目标。
- **可访问性**：Web 控件键盘可达、对比度达标、`alt` 文本（对标 PixelRefiner 的 a11y）。
- **国际化**：Web UI 中/英双语（用户 base 含中文用户）；CLI 错误信息可本地化（低优）。
- **离线**：Web 首次加载后可离线用（PWA / Service Worker 缓存 WASM）。
- **体积**：WASM 默认包 < 500KB（feature gate 关 EM/vectorize）；监控回归。
- **文档**：每个 CLI flag 有 `--help` 条目；Web 每个控件有 tooltip。
